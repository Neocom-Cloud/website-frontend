# NeoCom Website Frontend

Static multi-page marketing site for `neocom.cloud`, built with Vite and React and deployed to GitHub Pages.

This repository is `pnpm`-first. Use `pnpm` for install, development, testing, and build commands.

## Stack

- Vite
- React
- TypeScript
- Vitest + Testing Library
- GitHub Actions

## Local development

```bash
pnpm install
pnpm dev
```

## Scripts

```bash
pnpm generate:pages
pnpm typecheck
pnpm test
pnpm test:watch
pnpm build
pnpm test:build-output
BASE_REF=deploy HEAD_REF=main pnpm verify:promotion
pnpm preview
```

`pnpm generate:pages` regenerates the localized HTML entry points and `public/sitemap.xml` from the centralized content catalog.

## Architecture

- Shared React page models live in `src/App.tsx`.
- Localized copy and SEO strings live in:
  - `src/content/locales/pt-br.js`
  - `src/content/locales/en.js`
- Per-project non-translatable configuration lives in `src/content/projects.js`.
- Static localized HTML files and the sitemap are generated by:
  - `scripts/static-site.mjs`
  - `scripts/generate-pages.mjs`

This keeps one shared landing template, one shared standard project template, and one centralized translation source of truth.

## Asset policy

- Prefer `.svg` for logos, icons, and social preview assets whenever the source asset exists in SVG.
- Current SVG-first usage:
  - NeoCom brand: SVG
  - DevRecord: SVG
  - NeoRecicla: SVG
- Temporary exception:
  - Neo Health still uses PNG because there is no source SVG in the repository yet

## Locale behavior

- Root entry `/` redirects using this precedence:
  - `?lang=...` query override
  - stored locale from `localStorage`
  - browser locale from `navigator.languages`
  - fallback to `pt-br`
- Localized routes remain static for SEO:
  - `/pt-br/`
  - `/en/`
  - `/pt-br/projects/:slug/`
  - `/en/projects/:slug/`
- GitHub Pages cannot do true server-side locale detection from request headers. If you later need `Accept-Language` or geo-based edge routing, add that at Cloudflare or AWS.
